Building a Zero-Dependency UI Library

SL
Sarah Lin
March 28, 2026
8 min read

When we started building DevKit, we made one unconventional decision early on: no dependencies. Not a single npm package. This constraint shaped everything about the library -- from architecture to testing to how we think about long-term maintenance.

1200 x 630 -- Featured Image

Why zero dependencies?

The JavaScript ecosystem moves fast. Packages deprecate, APIs change, and supply-chain attacks are increasingly common. For a UI library that developers will rely on in production, we wanted to eliminate every point of external failure.

This does not mean we reinvented the wheel. It means we were deliberate about what we included and wrote exactly the code we needed -- nothing more, nothing less.

"The best dependency is the one you don't have. Every package you add is a bet that its maintainer will keep it working." -- DevKit Engineering Principles

The architecture

DevKit is structured as a layered CSS system with a thin JavaScript initialization layer. The CSS uses native @layer directives to manage specificity without resorting to excessive nesting or !important declarations.

css 
/* Layer order determines specificity */
@layer reset, tokens, base, layout, components, utilities;

/* Tokens define the design system */
@layer tokens {
  :root {
    --bg-root: #050505;
    --accent: #10b981;
    --font-sans: 'Inter', sans-serif;
  }
}

Component registration

The JavaScript side follows a simple registration pattern. Each component registers an initializer function that gets called on elements matching a data-dk-* attribute. A MutationObserver handles dynamically added elements automatically.

javascript 
// Register a component
DK.register('accordion', function(el) {
  var triggers = DK.$$('.dk-accordion_trigger', el);
  triggers.forEach(function(trigger) {
    DK.on(trigger, 'click', function() {
      // toggle logic
    });
  });
});

What we learned

Building without dependencies forced us to truly understand every line of code in the library. There are no black boxes. When something breaks, we know exactly where to look and how to fix it.

Here are the key takeaways from this approach:

  • Bundle size is predictable. Without transitive dependencies, the final size is exactly what we ship -- nothing hidden.
  • Debugging is straightforward. The stack trace always leads to our code, not into a maze of third-party modules.
  • Updates are on our schedule. We never have to rush a release because a dependency pushed a breaking change.
  • Security audits are simple. There is no supply chain to audit beyond our own repository.
Note
Zero dependencies does not mean zero tooling. We still use development tools like linters and test runners -- they just do not ship to your browser.

The tradeoffs

This approach is not free. Writing everything from scratch takes longer upfront. Features that would take minutes with a library take hours to implement correctly. But for a project meant to last years, the investment pays for itself many times over.

If you are building a component library, we encourage you to at least consider this approach. You might be surprised how little you actually need from external packages.
